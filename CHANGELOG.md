# Changelog

All notable changes to the Truto Agent Skills are documented here.

The format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html). Skills are content artifacts (not code), so versioning here tracks **observable changes to the guidance an agent receives** — added skills/references, restructured content, breaking removals, etc.

Dates are `YYYY-MM-DD`.

### Added

- `skills/truto-operator/SKILL.md` — New skill: **operator/debug playbooks for the in-dashboard Platform Assistant**, separate from the build-focused `truto` skill and the terminal-focused `truto-cli`. Frames the operator persona, documents the six admin meta-tools (`list_api_operations`, `describe_api_operation`, `call_platform_api` incl. `fields`/`jsonata`/`handle_id`, `query_tool_result`, `get_capabilities`, `read_platform_resource`), states the **adaptive contract** once (adapt to `route_context` / branch on evidence / stop when sufficient / never write on momentum / explain & cite), restates the grounding mandate, and captures the real approval reality (reads run immediately; writes/destructive need approval; the approval card shows only `METHOD /path` + a Write/Destructive badge — no body/diff/summary; `/assistant`, `/auth`, `/link-token`, `/magic-link` are forbidden). Includes the installed-vs-catalog rule (`GET /environment-integration` / `/integrated-account`, not `/integration`), the GET-list `environment_id` auto-inject, the rememberable-vs-hard-destructive distinction (`DELETE`/`bulk-delete` never remembered), and a "Not covered here" boundary (app code, integration authoring, Link SDK, workflows/daemon/batch). Ends with a routing table and a References table linking the seven playbooks + the shared error/evidence model.
- `skills/truto-operator/references/error-and-evidence-model.md` — Shared model ported so it's reachable at runtime (it isn't in the other bundled docs): the error envelope (`statusCode`/`error`/`message`), the `truto_is_remote_error` Truto-vs-provider split (`raw_response` + forwarded headers), every `truto_error_insight` key with the proxy-only-gets-`forbidden_error` caveat and the **`conditionally_required_*` = informational (schema-existence, not violation)** distinction, status-code semantics (incl. remote-401→`needs_reauth`, 405 sandbox read-only, 503 blocked), the **verified rate-limit tiers (per token 500 req/10s, per integrated account 50 req/10s, both `Retry-After: 10`)**, the retry/don't-retry rule, and the `/log` quick-map (the five log types with their **verified filter allow-lists** — incl. `mcp` `request_id`→`http_request_id` — vs the entry fields you read, plus the filter-vs-field distinction).
- `skills/truto-operator/references/triage-and-scope.md` — **P0**, the router: turn a vague report into surface + scope + symptom using `route_context` and one `get_capabilities` read, then branch to the right surface playbook. Skip when the user already named everything; never write here.
- `skills/truto-operator/references/safe-admin-changes.md` — **P1**, the write path every playbook hands off to: the verified read/safe-write/destructive/forbidden classifier (incl. `DELETE`/`*/bulk-delete` never-remembered, credential-touching bodies destructive), the minimal approval card and why you must explain the diff in words, scope selection (account vs environment vs base integration), read-current-state-first, endpoint merge semantics (mapping wholesale-replace, context shallow-merge, secret redaction), and doc-grounding the body before sending.
- `skills/truto-operator/references/debug-proxy-api.md` — **P2**: Truto-side vs provider-side via `truto_is_remote_error`, capabilities for resource existence/casing, `unified_proxy_api` logs (`request_type: proxy`), and a status-code branch table (401/403→P6, 404 casing, 405 sandbox, 429 self-heal, 5xx split). Supersedes `proxy-api` + `integration-troubleshooting` stubs.
- `skills/truto-operator/references/debug-unified-api.md` — **P3**: the hard-error vs data-shape fork; `truto_error_insight` keys; the 3-layer mapping (base / environment / account) and how to read the **merged** mapping via the meta endpoint; the **wholesale-replace gotcha** (`response_mapping` strings and arrays replace, not merge); branch to the narrowest layer via P1. Supersedes `mapping-changes` + the mapping half of `sync-debugging`.
- `skills/truto-operator/references/debug-sync-jobs.md` — **P4**: the six run states (incl. `rate_limited`, absent from the legacy run-state table), `resource_stats` as the stop-point, `rapid_bridge` logs, the **5-minute** stuck threshold, cron-miss diagnosis, and the no-reflexive-`ignore_previous_run` rule. Adds destination-type debugging for `completed`-but-empty runs (webhook → P5; datastore S3/GCS/Qdrant/Mongo via the `datastore_successful` log field + [Datastores]; SuperQuery has no outcome log field), and frames stuck recovery as a platform/admin concern (`/admin/sync-job/restart-stuck` exists but is outside the assistant's catalog). Supersedes the four-bullet `sync-debugging` stub.
- `skills/truto-operator/references/debug-webhook-delivery.md` — **P5**: exists/active/subscribed → attempted → accepted, reading `webhook_endpoint_status` to assign blame to the customer's endpoint, and `POST /webhook/test` (no approval). Extends the create-only `webhook-setup` stub with the delivery-debugging path.
- `skills/truto-operator/references/diagnose-integrated-account.md` — **P6**: the five-value account status enum, `last_error` / `last_forbidden_error`, the remote-401→`needs_reauth` lifecycle, and the status→reason→actor table — almost always an end-user reconnect (the assistant can't even mint a link token), with the one narrow sanctioned account write (`validation_error` context fix). New surface; the other playbooks branch here for anything auth-shaped.

  > Discoverability note: these playbooks are auto-bundled to `truto://skill/truto-operator[/references/{slug}]` from `main`, but the assistant's skill **index** and prompt routing come from the backend's hardcoded skills manifest. A follow-up backend PR must add a `truto-operator` manifest entry (and ideally repoint the thin `truto://guide/*` stubs) before the assistant will surface them. Until then they are reachable by URI but not discoverable.

- `skills/truto-unified-mappings/SKILL.md` — New skill for the agentic `truto unified-mappings build` workflow. Covers the full build → validate → apply loop that authors JSONata mapping rows connecting an integration's proxy API to a unified model (`crm`, `ats`, `hris`, …): one-time setup (profile keys, `unifiedMappingDir` / `integrationConfigDir`), live-grounded vs doc-grounded builds, the agentic loop vs `--structured` pipeline, the refinement loop, `validate` (deterministic audit, advisory warnings), `apply` (base vs env override rows, `--dry-run`), resume + `--only-missing`, and a key-flags reference.
- `skills/truto-unified-mappings/references/agent-and-tools.md` — The two build modes (default agentic loop with plan → route → sample → generate → commit → finalize, and the `--structured` fixed per-cell pipeline), the full agent tool list grouped by purpose (corpus, proxy + routing, sampling, generation, validation, finalize, server web tools), the live → DB → source → scrape sampling ladder, read-only live sampling (security), `--no-web-search`, prompt-caching/cost behavior (`agent.usage` log events), and the JSONL debug log.
- `skills/truto-unified-mappings/references/mapping-file.md` — The MappingFile JSON shape: file → database row mapping (base `unified_model_resource_method` vs env override), top-level fields, the `MappingCell` (`config`, `apply_db_action`, `sample_provenance`, `proxy_match`, `validation`), the per-cell `config` fields (`response_mapping`, `query_mapping`, `request_body_schema`/`request_body_mapping`, `error_mapping`), the `build_summary` review surface (`status` vs `outcome`, `skipped` vs `unbuilt_routed`, the 13 warning kinds, `counts.flagged`), `refinements`, and how `apply` consumes only `cells`.
- `skills/truto-integrations-build/SKILL.md` — New skill for the LLM-first `truto integrations build` workflow. Covers the full build → lint → apply loop: one-time setup (profile keys, `integrationConfigDir`), source-tier selection (OpenAPI / Postman / Mintlify / GraphQL / generic docs), Phase A (autonomous agentic build with 15 tools), Phase B (interactive refinement with `:edit`), the docs phase, `truto integrations lint` (static auditor with exit-code contract), and `truto integrations apply` (non-interactive push with `--dry-run` support). Key flags reference, iterating on existing integrations, docs-only mode, catalog awareness, troubleshooting and cost control knobs.
- `skills/truto-integrations-build/references/orchestrator-phases.md` — Deep-dive into the three build phases: Phase A (autonomous build, agent tools, patch format, `build_complete` requirements), Phase B (interactive refinement loop, `:edit` command), and the docs phase (per-method description / query_schema / body_schema generation, integration-wide readme / oauth_* rows, docs-only mode). Includes the full section order table.
- `skills/truto-integrations-build/references/integration-file.md` — IntegrationFile JSON shape: top-level fields (`name`, `config`, `id`, `label`, `category`, `documentation`, `audit_notes`), documentation row types (per-method `description` / `query_schema` / `body_schema`, integration-wide `readme` / `oauth_*`), audit note shape, how `apply` uses the file (parse → split → upsert → diff docs → summary), and tips for hand-editing.
- `skills/truto-integrations-build/references/lint-and-audit.md` — All six audit sources (`presence_check`, `critic_method_config`, `description_quality`, `method_coverage`, `pattern_match`, `method_naming`) with severity levels, example findings, and recommended actions. Covers the `method_naming` blocker rule (numbered suffixes + Issue 7 path-template check), exit-code contract (0 / 1 / 2), `--ignore-warnings` / `--ignore-info` semantics, structured output formats (`json` / `yaml` / `ndjson` / `csv`), file vs. slug input modes, and the `--integration-config-dir` resolution chain.
- `skills/truto-integrations-build/references/troubleshooting.md` — Source-tier selection guide (12 tiers with fidelity order), cache management (`~/.truto/cache/firecrawl/` 24h TTL, `~/.truto/cache/anthropic/` 7d TTL, `--refresh-*-cache`, `--no-llm-cache`), Firecrawl-tier gotchas (enterprise-only `--firecrawl-ignore-robots-txt`, fallback chain), cost control (`--max-pages`, `--no-firecrawl`, `--no-embeddings`, `--no-companion-docs`), stuck/slow build remediation, `--legacy-flow` escape hatch, and embedding error handling.
- `skills/truto/references/authoring-integrations.md` — End-to-end reference for authoring a brand-new integration definition (the customer-side equivalent of what Truto does for built-ins). Covers the full `integration.config` schema documented through the W1-typed OpenAPI components (`IntegrationConfig`, `IntegrationCredential`, `IntegrationResourceMethod`, `IntegrationPagination`, `IntegrationRateLimit`, `IntegrationWebhookConfig`, `IntegrationAction`, etc.), the recommended `truto integrations init` → `validate` → `create` workflow, all five credential formats (`api_key`, `oauth2`, `oauth2_client_credentials`, `oauth`, `keka_oauth`) plus the `basic` shortcut and BYOA semantics, the three authorization formats, the full per-method `IntegrationResourceMethod` field reference (including custom non-CRUD methods), the six pagination strategies, the inbound webhook receiver (`signature_verification` / `handle_verification` / `payload_transform`), the four reserved lifecycle actions (`post_install`, `validation`, `refresh_token`, `post_connect_user_form`) with a worked `post_install` example, and an end-to-end **Acme CRM** worked example (API-key auth, two resources with one read-only, an HMAC-signed webhook receiver, custom rate-limit detection). Closes the §3.2 audit gap; links into the typed OpenAPI components instead of duplicating the schema.
- `skills/truto/references/getting-started.md` — Day-1 tutorial that strings the `truto` CLI together with the application code: install → `truto login` → connect a sandbox → write the link-token route → first unified API call → port the same call into your app. Includes Express, Next.js Route Handler, and Hono / Cloudflare Workers variants of the link-token route.
- `skills/truto/references/customizing-integrations.md` — Per-environment integration overrides for the HTTP layer: authorization, pagination, rate-limit detection, and inbound webhook handling. Documents the new `truto environment-integrations override-*` CLI helpers, the JSONata scope variables for each surface, inspect/remove/test workflows, and a direct-HTTP appendix.
- `skills/truto/references/proxy-and-custom-api.md` — New "Authoring Custom-API Handlers" section covering both extension patterns: registered custom methods on a proxy resource (`integration.config.resources.{resource}.{methodName}`) and ad-hoc `/custom/{path}` calls with optional `methodConfig` body overrides.
